Проект посвящён разработке системы оптимизации логистики доставки,
основанной на задаче маршрутизации транспортных средств с временными
окнами и ограничениями по грузоподъёмности (CVRPTW), дополненной
отдельным ресурсом — грузчиками. Необходимо построить маршруты как для
транспортных средств, так и для грузчиков, учитывая временные окна заказов,
вместимость автомобилей, ограничения по длительности смен, необходимость
привлечения одного или нескольких грузчиков на отдельных заявках, а также
возможность отказа от выполнения необязательных заказов со штрафом.
Система должна определять оптимальные маршруты автомобилей и грузчиков,
синхронизировать их прибытие на точки доставки, учитывать различные скорости
передвижения и время выполнения работ, а также обеспечивать выполнение
всех обязательных заказов в заданные временные интервалы. Автомобили
начинают и заканчивают работу на складе, а грузчики начинают и завершают
смену в точке своего первого заказа.
Основной целью является минимизация совокупных затрат, включающих
использование транспортных средств, работу грузчиков, расход топлива,
длительность смен грузчиков и штрафы за невыполнение необязательных
заказов. При этом необходимо соблюдать ряд жёстких ограничений, связанных с
вместимостью автомобилей, временными окнами, последовательностью
маршрутов и требуемым количеством грузчиков на каждом заказе.
Входные данные представлены в формате JSON и содержат описание склада,
параметров транспортных средств и грузчиков, списка заказов, а также
коэффициентов целевой функции. Результатом работы алгоритма должен быть
JSON-файл с маршрутами автомобилей и грузчиков, отражающий
последовательность посещения заказов и времена начала выполнения операций.
Для реализации решения допускается использование существующих open-source
решателей задач маршрутизации, таких как OR-Tools, PyVRP или VRPSolverEasy,
с последующей адаптацией под особенности поставленной задачи.
Ключевые наблюдения
ИИ анализ задачи
Тип задачи
· CVRPTW с расширением (Capacitated Vehicle Routing Problem with Time Windows)
+ дополнительный ресурс — грузчики.
· Нужно построить маршруты одновременно для автомобилей и для грузчиков с
их синхронизацией.
Основные объекты и их особенности
· Склад – начало и конец маршрутов для автомобилей.
· Автомобили:
· Ограничения по вместимости (грузоподъёмность).
· Различная скорость передвижения.
· Начинают и заканчивают на складе.
· Грузчики:
· Начинают и завершают смену в точке первого заказа (не на складе).
· Ограничения по длительности смены.
· На некоторых заказах требуется один или несколько грузчиков.
· Заказы:
· Обязательные (должны быть выполнены) и необязательные (можно отказаться
со штрафом).
· Имеют временные окна доставки.
· Для каждого заказа известно требуемое количество грузчиков.
· Есть время выполнения работ (погрузка/разгрузка, учёт грузчиков).
Синхронизация
· Необходимо синхронизировать прибытие автомобиля и грузчиков на точку
доставки (они должны быть в одном месте одновременно для выполнения
заказа).
Ограничения (жёсткие)
· Соблюдение временных окон заказов.
· Вместимость автомобилей.
· Последовательность маршрутов (порядок объезда).
· Длительность смен грузчиков.
· Требуемое количество грузчиков на заказе.
· Обязательные заказы должны быть выполнены.
Целевая функция (минимизация совокупных затрат)
Включает:
· Использование транспортных средств (фиксированные или переменные
затраты).
· Работу грузчиков (стоимость человеко-часов или смен).
· Расход топлива (зависит от пройденного пути).
· Длительность смен грузчиков.
· Штрафы за невыполнение необязательных заказов.
Входные данные
· Формат JSON.
· Содержит: описание склада, параметры ТС, параметры грузчиков, список
заказов, коэффициенты целевой функции.
Результат
· JSON-файл с маршрутами автомобилей и грузчиков:
· Последовательность посещения заказов.
· Время начала выполнения операций на каждом заказе.
Технологические ограничения на решение
· Допускается использование open-source решателей: OR-Tools, PyVRP,
VRPSolverEasy.
· Нужна адаптация под специфику задачи (синхронизация двух типов ресурсов,
начало грузчиков не со склада и т.д.).
Готовые решения
· Яндекс Маршрутизация и «ЯКурьер»: Российские системы для построения
маршрутов доставки.
· LogiNext и eLogii: Известные зарубежные платформы для оптимизации
«последней мили».
· Класс TMS (A.TMS, Axelot): Комплексные системы управления перевозками.
ИИ анализ инструментов
🛠 OR-Tools (от Google)
Это универсальный «швейцарский нож» для задач комбинаторной оптимизации
(CVRP, VRPTW, задача о ранце, планирование и др.). В OR-Tools можно «руками»
через API описать почти любые ограничения, но за это приходится платить
большим объёмом ручной работы и потенциально низким качеством решений для
некоторых вариантов VRP по сравнению со специализированными
инструментами.
🧬 PyVRP
Это узкоспециализированный «спринтер» для CVRP, VRPTW и Prize Collecting,
который даёт очень высокое качество решений на больших объёмах данных.
Однако новые типы ограничений (вроде синхронизации с грузчиками) придётся
реализовывать самостоятельно, так как их нет «из коробки».
🧮 VRPSolverEasy
Инструмент для тех, кому нужно точное решение (доказуемый оптимум для
сложных rich VRP, до ~250 точек). Однако он очень требователен к ресурсам и
чувствителен к размерам: если заказов станет больше 100–250, решение может
затянуться на часы.
Оценка готовых решений относительного нашего проекта
1. OR-Tools (Google)
Основной функционал:
Универсальный фреймворк для решения широкого спектра задач оптимизации,
включая сложные VRPTW. Позволяет вручную прописывать практически любые
ограничения через API, имеет обширную документацию на нескольких языках.
Сильные и слабые стороны:
• ✅ Гибкость: Позволяет реализовать любые кастомные ограничения — от
временных окон до синхронизации.
• ✅ Поддержка: Активное сообщество, множество примеров (CVRP, VRPTW,
Resource Constraints).
• ❌ Качество на больших данных: На инстансах выше 500–1000 точек
качество решений падает по сравнению с PyVRP и VROOM.
• ❌ Сложность разработки: Для реализации нетривиальной логики
грузчиков требуется высокая квалификация.
• ❌ Язык: Чистый C++ для производительности, Python-обертка для
моделирования.
2. PyVRP
Основной функционал:
Современный, высокопроизводительный решатель на Python/C++ от RoutingLab,
ориентированный на поиск решений, близких к оптимальным, с помощью
эвристики "разрушай и воссоздавай".
Сильные и слабые стороны:
• ✅ Качество решений: Лучшая точность среди аналогов на CVRP
(отставание ~0.23% от лучшего известного решения).
• ✅ Масштабируемость: Уверенно решает инстансы с тысячами визитов.
• ✅ Поддержка сложных вариантов: Включает поддержку мульти-поездок
(Multi-trip), альтернативных визитов, релоадинга.
• ❌ Ограничения: Не поддерживает Shipments (перевозки типа Pickup-
Delivery).
• ❌ Ради точности: Требует длительного времени счета (минуты-часы) для
достижения наилучших результатов.
3. jsprit
Основной функционал:
Легковесный Java-инструмент с отличной реализацией паттерна Builder,
позволяющей буквально собирать модель задачи из готовых компонентов.
Сильные и слабые стороны:
• ✅ Простота и понятность: Java-инструмент с интуитивным Builder-
паттерном, что ускоряет старт разработки.
• ✅ Удобство Java-стэка: Легко встраивается в корпоративные приложения.
• ❌ Производительность: Показывает худшее качество решений и
испытывает трудности на больших масштабах (>500 точек).
• ❌ Язык: Проект на Java требует Python-обертки для интеграции.
4. VROOM
Основной функционал:
Высокопроизводительный C++ движок, ориентированный
на молниеносное решение задач (миллисекунды) за счет сбалансированного
подхода к точности.
Сильные и слабые стороны:
• ✅ Скорость: Решает сложные задачи за миллисекунды, что идеально для
интерактивных систем и сценариев перепланирования.
• ✅ Богатство моделей: Поддерживает MDHVRPTW, PDPTW, работу с
навыками, рабочими часами и перерывами водителей.
• ❌ Качество решений: Жертвует оптимальностью ради скорости, давая
"хорошие", но не всегда лучшие решения.
• ❌ Синхронизация: Не поддерживает несколько типов ресурсов
одновременно.
5. VRPSolverEasy
Основной функционал:
Питоновский интерфейс для точного решателя VRPSolver, основанного на
алгоритме Branch-Cut-and-Price.
Сильные и слабые стороны:
• ✅ Точность: Способен находить доказуемо оптимальные решения для
сложных вариантов задачи.
• ✅ Простота моделирования: Абстрагирует пользователя от МIP-
моделирования через интуитивную структуру данных.
• ✅ Поддержка "Rich VRP": Позволяет добавлять мульти-депо,
опциональных клиентов, альтернативные временные окна и
несовместимости.
• ❌ Масштаб: Практичен для задач до 100 клиентов, на 200+ требуется
мощный сервер и долгое время счета.
• ❌ Пригодность: Предназначен для R&D (НИОКР) и не подходит для
production-среды из-за отсутствия стабильности и коммерческой
поддержки.
